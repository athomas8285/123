## 六、每日操作流程

```
步骤 1: 获取竞彩网比赛（控制台按钮 "获取竞彩网比赛"）
   ↓  竞彩网 API → fetch_jczq.py → raw_jczq.json
   ↓  弹窗勾选确认 → confirm_jczq_matches API → 写入 DB + match_info.json + locked_data.json
   ↓
步骤 2: 运行预测（控制台按钮 "运行预测" — 勾选待预测比赛后点击）
   ↓  run_all.py --match-ids ≡ λ计算 → 因子修正 → MC模拟 → DDI校准 → 贴合度评分 → 评级判定
   ↓  写入 rating_result.json + monte_carlo_result.json + ddi_result.json + fit_score_result.json + DB
   ↓
步骤 3: 生成计划池（控制台按钮 "生成计划池" — 可以步骤 2 完成后自动触发）
   ↓  gen_plan.py → plan_data.json
   ↓  gen_plan_html.py → static/plan.html
   ↓  gen_featured.py → featured_data.json
   ↓  gen_featured_html.py → static/featured.html
   ↓
步骤 4: 查询赛果（控制台按钮 "一键查询赛果"）
   ↓  500.com → fetch_results.py → 写入 DB actual_score + half_full + 自动计算 hit
   ↓  赛果弹窗显示结果，关闭后自动跳转基本面面板
   ↓
   → 基本面分析 / 计划池 / 精选计划单 自动刷新
```

**流程可反复执行**：步骤 2-4 对同一批比赛可多次运行。

## 八、计划池 — 腿筛选规则（gen_plan.py V5.3）

### 8.1 腿类型

| 类型 | 数据源 | 说明 |
|------|--------|------|
| **方向（单腿）** | 竞彩 SP 赔率 + 让球赔率 | 主胜 / 平局 / 客胜 / 让胜 / 让平 / 让负，共 6 个 |
| **方向打包** | 合并两档赔率 | 胜/平、负/平、让胜/平、让平/负，虚拟赔率 = `1 / (1/o1 + 1/o2)` |
| **半全场（单腿）** | MC top2 半全场预测 | 从 hafu 赔率取对应 key，共 2 个 |
| **半全场打包** | 合并 MC top2 双项 | 主不败(胜胜+平胜) / 客不败(负负+平负) |

### 8.2 方向兼容矩阵

- 预测方向为 **胜/让胜**：排除 `负`、`客胜`、`让负`（仅保留胜侧）
- 预测方向为 **负/让负**：排除 `胜`、`主胜`、`让胜`（仅保留负侧）
- 预测方向为 **平**：全部通过
- **让平** 始终通过（和所有方向兼容）

### 8.3 显示层 handicap 替换

当 `|handicap| = 1` 时：
- `hcp = -1`：方向打包(让胜/平) 用 `display_option=主胜` 替换显示（赔率更高）
- `hcp = +1`：方向打包(让平/负) 用 `display_option=客胜` 替换显示
- 替换仅在方向单腿赔率 > 虚拟赔率时生效
- **不影响排序和组合产生**，仅影响 HTML 渲染

### 8.4 组合排序规则

每条腿必须来自不同比赛（`l1.mid ≠ l2.mid`）。组合生成后排序：
1. **先按赔率接近 2.0**：`abs(raw_opt_odds - 2.0)` 升序
2. **再按保底回报**：`min_odds` 降序
3. **再按 EV**：`type_hitrate × avg_payout` 降序

### 8.5 档位划分

| 档位 | 赔率区间 | 取前 N | 说明 |
|------|---------|--------|------|
| 2.0 计划单 | 1.9 ~ 2.2（逐步放宽到 2.5） | TOP 5 | 低风险，追求稳定回报 |
| 3.0 计划单 | 2.5 ~ 5.0 | TOP 5 | 中高风险，追求较高回报 |

## 九、精选计划单 — 追回策略（gen_featured.py）

每个比赛日从计划池的 **2.0 档第 1 名** 作为当日精选：

| 参数 | 值 |
|------|-----|
| 基础投入 | ¥1,000 |
| 追回倍数 | [1, 1, 2, 4, 8, 16] |
| 最大轮次 | 6 轮 |
| 命中判定 | 两腿都命中（`judge_leg_hit` 逐腿校验） |
| 命中后 | 轮次归零，倍数重置为 1 |
| 偏离后 | 轮次 +1，下期倍数递增 |
| 累计盈亏 | 每期 `profit = odds × stake - stake`（命中）/ `profit = -stake`（偏离） |

`judge_leg_hit` 逻辑覆盖：方向/方向打包/半全场/半全场打包四种腿类型，根据 `actual_score` + `half_full` + `handicap` 判定。

## 十二、编码约定

### 12.1 中文处理 → 极其重要

- **禁止** `@"...""@` PowerShL 管道方式执行含中文 Python 代码 → 中文变 `?`
- **必须** 写独立 `.py` 文件后 `python file.py` 执行
- 所有 `.py` / `.js` / `.html` 文件存为 **UTF-8**
- `.js` 文件中的中文比较字符串（如 `"胜"`、`"负"`）极易因编码变 `?` 导致 hit 永远算不出

### 12.2 重启流程

```bash
pkill gunicorn
rm -rf routes/__pycache__ static/panels/__pycache__
```
